# restaurant-voice-order-saas

SaaS MVP for restaurant voice ordering, built with Spec-Driven Development and an AI harness.

## Product intent

The product enables restaurant waiters to speak orders into an Android app, review the structured draft generated by STT/parser components, edit or reject it when needed, and only then submit confirmed order items to kitchen/bar screens in real time.

The core safety rule is mandatory: AI-generated or AI-parsed orders are drafts only and must never be submitted, persisted as submitted, or emitted to kitchen/bar screens before explicit human waiter confirmation.

## Selected MVP stack

- Android waiter app: Flutter.
- Web admin/kitchen/bar: Flutter Web for MVP.
- Backend API: Python FastAPI.
- Database: PostgreSQL.
- Realtime: authenticated WebSocket.
- AI/voice: STT and parser adapters behind application interfaces.

## Monorepo structure

```text
  web_console/      # Flutter Web admin + kitchen/bar console, skeleton only in T-001

  api/              # Python FastAPI backend with health endpoint and local env example
packages/
  api_contracts/    # Shared API contract artifacts, skeleton only in T-001

```


## Architecture responsibility boundaries

In this repository, `services/api` is the folder for the deployable backend API service. Inside that backend, implementation must still be separated by responsibility: domain rules stay in `domain`, application workflows stay in use cases, external protocol/provider code stays in gateways/adapters, and technical wiring stays in infrastructure.

No code should mix these contexts. For example, an HTTP route or WebSocket gateway must call an application use case instead of implementing order-confirmation logic directly, and STT/parser adapters must never persist submitted order items or emit kitchen/bar events.
